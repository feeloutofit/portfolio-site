문제

리뉴얼 버전은 기존 골프몬 서비스와 같은 정책으로 운영되어야 했지만, 실제 예약 흐름, 관리자 업무, 데이터 처리 방식 중 일부가 누락되거나 다르게 구현된 상태였습니다.

화면에서 보이는 오류만 고치면 되는 상황이 아니었습니다. 예약/ERP 연동, 양도·조인, 정산/실적, 인증/로그인, 앱 WebView와 푸시까지 연결되어 있어 기존 PHP 웹/관리자, 리뉴얼 백엔드/프론트, Android/iOS 앱의 흐름을 함께 비교해야 했습니다.

대표 해결 사례

예약/연동 자동 마감과 중복 등록 방지

리뉴얼 버전에서 예약, 티타임, 양도·조인 상태 변경이 기존 운영 정책과 다르게 동작하는 구간을 확인했습니다. 기존 서비스의 마감 기준, 중복 등록 조건, 관리자 처리 흐름을 비교한 뒤 자동 마감과 중복 방지 로직을 보완했습니다.

  • 예약/티타임 연동, 취소·변경 조건, 골프장별 노출 정책을 함께 확인했습니다.
  • 사용자 화면, 관리자 화면, 데이터 저장 결과가 같은 상태 기준을 보도록 검증했습니다.

관리자 세션 이력과 중복 로그인 제한

운영 계정은 여러 사람이 사용하거나 같은 계정으로 동시에 접근할 때 처리 주체가 흐려질 수 있었습니다. 관리자 세션 이력과 동시 접속 제한 흐름을 보완해 로그인, 디바이스 제한, 세션 만료 조건을 운영 기준에 맞게 정리했습니다.

  • 운영자가 문제 발생 시 접속 이력을 추적할 수 있도록 계정 사용 흐름을 남겼습니다.
  • SMS 인증, 인증 횟수 초기화, 중복 로그인 정책을 관리자 업무 흐름과 함께 점검했습니다.

외부 API, 배치, 앱 이슈의 원인 분리

외부 연동과 앱에서 발생하는 현상은 한 코드베이스만 보면 원인이 분명하지 않은 경우가 많았습니다. API 타임아웃, deadlock 방어, cron 중복 실행 방지, 앱 WebView/푸시/버전 이슈를 백엔드 응답과 관리자 설정까지 연결해 확인했습니다.

  • 장애 가능성이 큰 연동과 배치 작업에는 타임아웃, 예외 처리, 중복 실행 방지 기준을 보강했습니다.
  • Android/iOS 앱 이슈는 앱 설정, WebView 동작, 푸시, 분석 SDK와 API 응답을 함께 좁혀 봤습니다.

담당 범위

  • 기존 PHP 웹/관리자와 리뉴얼 Spring Boot/Vue.js 서비스의 기능 차이를 비교했습니다.
  • 예약/ERP 연동, 양도·조인, 국내투어, 1박2일 상품, 정산/실적, 인증 흐름을 보완했습니다.
  • 관리자 검색/필터, 예약 확인, 일괄 등록, 연동 호출 이력, 업무지원/QA, 업데이트 공지 화면을 개선했습니다.
  • 기존 Android 앱과 리뉴얼 Android/iOS 앱의 푸시, WebView, 앱 버전, 분석 SDK, 제스처 이슈를 운영 반영했습니다.

판단 방식

  1. 화면 오류를 바로 고치기 전에 기존 서비스의 실제 동작과 운영 담당자의 업무 기준을 먼저 확인했습니다.
  2. 단순 구현 오류와 정책 확인이 필요한 항목을 나누어 수정 범위와 우선순위를 정했습니다.
  3. 사용자 화면, 관리자 화면, 백엔드 데이터 흐름에 나뉜 변경을 하나의 예약/운영 흐름으로 묶어 검증했습니다.
  4. 운영 영향이 큰 예약, 정산, 인증 기능은 수정 후 관련 케이스를 다시 확인했습니다.
  5. 앱에서 보이는 현상도 WebView, 푸시, 앱 버전, API 응답, 관리자 설정을 함께 확인해 원인을 좁혔습니다.

결과

리뉴얼 산출물의 기능 누락과 운영 정책 불일치를 보완해 기존 골프몬 서비스 흐름과의 차이를 줄였습니다. 외부 산출물을 단순 인수하는 데서 끝내지 않고, 실제 운영에서 반복되는 오류와 예외 케이스를 기준으로 백엔드 로직, 화면 조건, 데이터 처리 흐름을 점검했습니다.

  • 리뉴얼 버전이 기존 예약/관리자 업무 기준에 맞게 동작하도록 누락 흐름을 보완했습니다.
  • 사내 QA 게시판 접수 건이 줄어드는 흐름을 확인했고, 사용자 규모가 커진 상황에서도 운영 반영을 이어갔습니다.
  • 백엔드, 사용자 화면, 관리자 화면, 앱 이슈를 하나의 서비스 흐름으로 추적하는 운영 안정화 경험을 쌓았습니다.
